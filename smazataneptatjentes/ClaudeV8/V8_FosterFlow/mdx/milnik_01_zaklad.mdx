## 0. Co tento milník obsahuje a proč

Milník 1 (M1) je rozdělen na 15 fází. Cílem není "udělat trochu ze všeho", ale postavit **úplné, bezpečné, zálohovatelné jádro** pro jednu organizaci — bez multi-tenancy komplexity, bez AI, bez pokročilých financí. Tyto věci přijdou v dalších milnících a NESMÍ zpomalovat cestu k bezpečnému ostrému nasazení.

<CardGroup cols={2}>
  <Card title="Co JE v M1" icon="check">
    Databázové jádro, RBAC, RLS, Karta kontaktu s historizací, sociální prostor, event store, dokumenty (bez OCR), dva produkty (Worker + Organization), konfigurovatelné tabulky, audit, zálohy + DR test, observabilita, accessibility baseline, i18n architektura, bezpečnostní audit, export dat.
  </Card>
  <Card title="Co NENÍ v M1 (záměrně odloženo)" icon="circle-xmark">

  </Card>
</CardGroup>

<Info>
**Proč SSO (přihlášení přes Google/Microsoft) NENÍ v M1:** je to explicitní rozhodnutí. M1 používá pouze interní autentizaci (e-mail + Magic Link). Architektura je připravena (viz Fáze 1.1, `AuthAdapterInterface`) tak, aby SSO šlo přidat v M5 BEZ refaktoringu, ale implementace SSO čeká, protože není na kritické cestě k bezpečnému provozu a každá OAuth integrace navíc je riziko a komplexita navíc před prvním ostrým nasazením.

### Provider-agnostické kontrakty — POVINNÉ od první řádky kódu

<Warning>
Toto je nejdůležitější architektonické pravidlo celého projektu. Žádný modul, žádná komponenta, žádná API route nesmí nikdy přímo volat Google API, Microsoft API, nebo konkrétního AI poskytovatele. Vše jde přes interní rozhraní definovaná zde v M1, i když M1 implementuje jen jeden ("internal") adaptér pro každé.
</Warning>

Definuj tato čtyři TypeScript rozhraní v `src/lib/adapters/`:

```typescript
// src/lib/adapters/StorageAdapterInterface.ts
export interface StorageAdapterInterface {
  uploadFile(orgId: string, file: Buffer, metadata: FileMetadata): Promise<StorageReference>;
  downloadFile(reference: StorageReference): Promise<Buffer>;
  deleteFile(reference: StorageReference): Promise<void>;
  getSignedUrl(reference: StorageReference, expiresInSeconds: number): Promise<string>;
}
// M1 implementace: InternalSupabaseStorageAdapter (jediná, používá Supabase Storage)
// M5 doplní: GoogleDriveAdapter, OneDriveAdapter — KONTRAKT SE NEMĖNÍ

// src/lib/adapters/CalendarAdapterInterface.ts
export interface CalendarAdapterInterface {
  createEvent(orgId: string, event: CalendarEventInput): Promise<ExternalEventRef>;
  updateEvent(ref: ExternalEventRef, event: CalendarEventInput): Promise<void>;
  deleteEvent(ref: ExternalEventRef): Promise<void>;
}
// M1 implementace: InternalCalendarAdapter (vlastní tabulka, žádný externí kalendář)
// M5 doplní: GoogleCalendarAdapter, MicrosoftGraphCalendarAdapter

// src/lib/adapters/AuthAdapterInterface.ts
export interface AuthAdapterInterface {
  initiateLogin(email: string): Promise<void>; // Magic Link
  verifyCallback(token: string): Promise<UserSession>;
}
// M1 implementace: InternalMagicLinkAdapter
// M5 doplní: GoogleOAuthAdapter, MicrosoftOAuthAdapter

// src/lib/adapters/AIAdapterInterface.ts
export interface AIAdapterInterface {
  generateText(prompt: string, context: AIContext): Promise<string>;
  transcribeAudio(audioBuffer: Buffer): Promise<string>;
}
// M1: rozhraní existuje, ŽÁDNÁ implementace, ŽÁDNÉ volání - AI se aktivuje až M6
```

Každý adaptér se vybírá runtime podle `organization_integration_settings` (tabulka existuje od M1, viz schema.sql sekce Provider-agnostická architektura). V M1 má každá organizace jen řádky s `provider='internal'`.

  <Accordion title="organizations, profiles, RBAC (permissions/roles/user_roles)">

  </Accordion>

  </Accordion>
</AccordionGroup>

## 3. Fáze 1.3 — Hierarchické registry institucí a historizace Karty kontaktu

### Tříúrovňová hierarchie institucí

Obecný vzor pro úřady, školy i zdravotnictví:

```
Úroveň 1: Zřizovatel (Kraj / Obec s rozšířenou působností)
   └─ Úroveň 2: Konkrétní pobočka/útvar (OSPOD Teplice / ZŠ Teplická / Poliklinika Teplice)
        └─ Úroveň 3: Konkrétní osoba (sociální pracovnice / třídní učitel / lékař)
```

Implementováno tabulkami `administrative_authorities` → `ospod_offices`/`schools`/`medical_facilities` → `ospod_workers`/`school_teachers`/`doctors`. Soudy mají analogickou strukturu (`courts` → `court_judges`).

### Historizace — nic se nepřepisuje, vše se historizuje

<Warning>
Dítě v pěstounské péči prochází systémem dlouhodobě, často od narození do zletilosti. Jméno, škola, lékař i adresa se v čase mění. Systém **nesmí** tyto údaje přepisovat — musí udržovat kompletní historii (od kdy – do kdy) pro účely soudních zpráv a auditů.
</Warning>

Tabulky: `person_past_names`, `person_citizenships`, `person_residency_history`, `person_contacts`, `person_address_history`, `person_education_history`, `person_report_cards`, `person_medical_history`, `person_physiological_metrics`, `person_identity_documents`, `person_consents`, `person_guardians`, `person_photo_documentation`.

### Vícejazyčnost (i18n) — architektura od první obrazovky

<Info>
M1 obsahuje **pouze** českou (`cs-CZ`) lokalizaci obsahu. Ale žádný uživatelsky viditelný text nesmí být natvrdo v React komponentě — vše jde přes `translation_keys`/`translation_values`. Přidání dalšího jazyka v budoucnu (slovenská DO, cizojazyčný biologický rodič) je pak čistě datová operace, ne refaktoring.
</Info>

## 4. Fáze 1.4 — Sociální prostor a vztahy (symetricky dítě/pěstoun)

<Warning>
Svěření do pěstounské péče se váže na konkrétní fyzickou osobu pěstouna, NE na rodinu jako celek — i když pěstoun žije v manželství/partnerství. Pěstoun má (nebo nemá) manžela/manželku, partnera/partnerku, vlastní biologické děti a svůj vlastní sociální prostor — to vše musí být v systému evidovatelné stejně důkladně jako sociální prostor dítěte.
</Warning>

Jedna sdílená tabulka `person_relationships` (NE dvě oddělené pro dítě a pěstouna) s `subject_id` (dítě NEBO pěstoun) a `related_person_id`. Typ vztahu je dynamický číselník (`relationship_type_term_id` → `taxonomy_terms`), seed obsahuje výchozí termíny pro obě strany (otec/matka/dědeček... pro dítě, manžel_pěstouna/biologické_dítě_pěstouna... pro pěstouna).

### Rating bezpečnosti vztahu

`relationship_rating` ENUM A–Z, kde `Z` = zákaz styku nařízený soudně/OSPODem. UI musí při ratingu `Z` zobrazit červený banner ve Worker aplikaci.

### Typ pěstounské péče A/B/C

```
A = Zprostředkovaná pěstounská péče (Dlouhodobá)
B = Pěstounská péče na přechodnou dobu (Přechodná)
C = Nezprostředkovaná pěstounská péče (Příbuzenská)
```

Zobrazuje se v závorce za příjmením (`Barbora Suchá (A)`), s tooltipem na hover.

## 5. Fáze 1.5 — Event store (configuration-driven)

Centrální tabulka `events` jako chronologická časová osa rodiny. `event_definitions` definuje JSON Schema pro každý typ události — frontend dynamicky generuje formulář, backend validuje payload před uložením (knihovna Ajv).

```sql
-- events.type je TEXT validovaný proti event_definitions.event_type,
-- NIKDY pevný ENUM - nový typ události = administrativní úkon, ne kódová změna
```

`post_actions` JSONB pole na `event_definitions` umožňuje "Workflow Lite" — po uložení události se automaticky vytvoří úkol nebo notifikace, bez nutnosti plnohodnotného workflow enginu (vědomě zjednodušený kompromis, viz historie rozhodnutí v Registru rozhodnutí).

<Check>
**Acceptance test Fáze 1.5:** Vytvoření události typu `visit_log` s chybějícím povinným polem `ai_summary` vrátí HTTP 400 s popisem chyby, data se NEULOŽÍ. Validní událost se zapíše a zobrazí na timeline rodiny.

## 6. Fáze 1.6 — Lifecycle state machine

`lifecycle_state` ENUM pro `households` (lead → initial_contact → negotiation → pending_orp → active_accompanying → situation_change → termination → archived) a `foster_parent_status` ENUM pro `persons` (zajemce → cekajici → aktivni → pozastavene / vyrazene / prevedene / odmitnute) — DVĖ ODDĖLENÉ stavové stroje, protože jeden pěstoun může mít víc dohod v různých fázích.

Přechod je možný jen po splnění podmínek (povinný dokument, vyplněné pole) a generuje záznam do `lifecycle_transitions`/`foster_parent_status_history` automaticky přes trigger.

<Check>
**Acceptance test Fáze 1.6:** Pokus o přechod rodiny z `pending_orp` do `active_accompanying` bez nahraného dokumentu Souhlas ORP je zablokován v UI (tlačítko disabled) a backend ho odmítne i při přímém API volání.
</Check>

</Info>
